# Design: inventory-items

## Context

物品頁目前為靜態顯示：`Equipment`（name/nameEn/type 自由字串/description/equipped）由創角流程帶入，無數量、無來源區分、無任何編輯入口；武器另存於 `weapons` 清單並驅動行動頁攻擊列。財富已有 `Currency`（pp/gp/ep/sp/cp）與計算機（`adjustCurrency` 批次增減、各幣別夾 ≥0）。內容庫（外部 Supabase 內容專案）已保留 `items`/`v_items` 表但尚未收錄資料；App 端 `CatalogRepository` 已有其他資料域（spells/classes/…）的查詢與快取模式可循。

## Goals / Non-Goals

**Goals:**
- 物品模型支撐跑團高頻操作:買、用、裝、丟、記任務物品
- 類型與來源正交:任何來源的物品都能是任何類型;任務旗標再正交於兩者
- 內容庫裝備目錄(SRD 5.2 範圍)可瀏覽/搜尋,購買自動扣款
- 舊角色資料無痛升級(JSON 向後相容,不需資料修復)

**Non-Goals:**
- 負重/攜帶容量計算
- 武器熟練精確判定（MVP 一律視為熟練加成，職業武器熟練後續提案接上）
- 雙手/單手切換（versatile）與雙持規則的傷害差異呈現
- 販售（賣出）流程與價格協商——本次僅購買；賣出後續提案
- Campaign 共用物資（屬 campaign-notes 範疇）

## Decisions

### D1. 擴充 `Equipment` 而非另立新模型
沿用既有 `Equipment` freezed 模型加欄位，全部帶預設值：
```dart
ItemType itemType;      // weapon / armor / gear / consumable，@Default(ItemType.gear)
ItemSource source;      // catalog / custom，@Default(ItemSource.custom)
int quantity;           // @Default(1)
bool quest;             // @Default(false)
int priceCp;            // 單價（銅幣計），@Default(0)；購買時寫入，供未來販售
String catalogRef;      // source=catalog 時的目錄鍵（engName），供詳情回查
```
既有 `type` 自由字串保留為顯示用副標（如「輕武器」），不參與行為判斷。
- 替代方案：新開 `InventoryItem` 模型並遷移——改動面大、舊資料要轉換，且 `equipment` 欄位名已同步到雲端 JSON，不值得。

### D2. 任務物品是「玩家自行標記」的旗標，不是類型也不是來源
Lorebook 架構上無從得知一件物品是否為任務物品——quest 是玩家宣告的記帳標記（與 `equipped` 同性質），系統不做任何推斷。其意義完全在於標記後換到的行為保護：刪除需二次確認、消耗品歸零保留、未來販售流程排除、清單徽章。`quest: true` 可疊加在任何類型與任何來源上（DM 指定一把 SRD 目錄裡的正規長劍為任務關鍵同樣成立）。
- 替代方案 A：quest 作為第五種類型——會讓「任務武器」無法表達，否決。
- 替代方案 B：quest 作為第三種來源——任務物品多為自訂輸入但不必然（目錄物品也可以是任務物品），且來源描述的是「資料怎麼來」而非「怎麼被對待」，否決。

### D3. 目錄資料走既有 CatalogRepository 模式
`v_items` 依 catalog-source 政策過濾（`source='XPHB'`、srd 範圍），App 端比照 spells 的 provider/快取模式新增 items 查詢：類別（武器/護甲/冒險裝備/工具）+ 關鍵字搜尋。價格在內容庫以 cp 整數存（`price_cp`），避免浮點與幣別字串解析。

### D4. 取得與付費解耦；成交金額可改；扣款採分層扣款、不重排幣別組合
「從目錄挑選」與「付費」是獨立的兩件事——同一瓶治療藥水可能是商店購買（扣款）、隊友給予或戰利品搜刮（皆不扣款），因此目錄詳情提供對等的「購買／直接取得」雙入口，扣款只發生在明確選擇購買時。

購買時 SRD 標價僅作**預填值**，成交金額可於確認前修改（DM 自訂價格、議價、折扣是跑團常態）；`priceCp` 存實際成交價，SRD 標價隨時可經 `catalogRef` 回查，兩者不混淆。

價格以 cp 計。扣款規則：
1. 換算玩家總財富為 cp 總值，低於成交金額 → 擋下並提示差額（玩家可改金額重試或改用直接取得）
2. 足夠 → 自幣值小到大先扣同額（cp→sp→ep→gp→pp），某階不足時向上一階「換零」（1 gp=10 sp、1 ep=5 sp、1 sp=10 cp、1 pp=10 gp），只換所需最小枚數
3. 寫回沿用 `adjustCurrency` 的增減語意
- 替代方案 A：只扣標價幣別、不足就擋——玩家帶 2 gp 買 15 sp 物品會被誤擋，體驗差。
- 替代方案 B：全部換算成 cp 重新配置——會摧毀玩家的幣別組合（跑團者在意「我有幾金幾銀」），否決。
- 替代方案 C：金額不足允許透支為負——與 Currency 各幣別 ≥0 的既有不變量衝突，且現實桌面也不會這樣記帳，否決。

### D5. 消耗品用完即移除
`consumable` 使用一次 `quantity - 1`，歸零即從清單移除（quest 消耗品例外：歸零保留為 0，避免劇情物件無聲消失）。移除時 SnackBar 提供復原。

### D6. 裝備效果：AC 與攻擊列改為推導值
**AC**：`ac` 靜態欄位棄用，改由純函式 `computeAc(character)` 推導：
- 無護甲：10 + Dex 調整值；野蠻人 10+Dex+Con、武僧 10+Dex+Wis（SRD 無甲防禦特性，依職業判定）
- 裝備護甲：依目錄 AC 公式參數（輕甲 base+Dex、中甲 base+Dex(上限2)、重甲固定值）
- 盾牌另計 +2，可與護甲疊加
- 舊資料遷移：讀入時若推導所需資料完整即用推導值；`ac` 欄位保留一版作為推導失敗的顯示 fallback

**裝備狀態判定**：三個純函式判定式（著甲中／著重甲中／持盾中）為 AC 與特性提示的共用基礎。SRD 無甲條件有三種嚴格度：野蠻人無甲防禦持盾仍有效；武僧無甲防禦/武術/無甲移動須無甲且無盾；野蠻人快速移動僅排除重甲。條件對照採 App 內建固定清單（SRD 範圍小而明確），不做通用條件引擎——特性資料本身沒有結構化條件欄位，硬做通用引擎是過度設計。

**法師護甲**：同用「著甲中」判定，但屬法術效果而非職業特性。完整效果追蹤（持續時間、來源管理）屬暫緩中的 active-effects 範疇；本次以輕量布林開關實作即可成立，因為 2024 規則「著甲即提前結束」讓它不需要倒數——裝甲時自動關閉開關（真結束，非失效提示），卸甲不自動恢復。未著甲時多個基礎公式（一般無甲/無甲防禦/法師護甲）取最高值。開關僅對法術清單含法師護甲的角色顯示。

**特性提示**：條件不滿足的特性在清單上標「著甲中失效」，不隱藏不刪除（玩家需要知道特性還在、只是暫時失效）。速度加成類特性（無甲移動 +10 呎等）本次僅做失效提示，速度數值推導另案（speed 目前是靜態字串，推導需職業等級表，範疇再擴就失控）。

**攻擊列**：靜態 `weapons` 清單退場，攻擊列 = 裝備中武器（itemType=weapon 且 equipped，機制資料來自目錄或自訂輸入）＋固定一列「徒手攻擊」（2024 規則全角色皆有：命中＝力調+熟練、傷害＝1+力調 鈍擊）。舊 `weapons` 資料讀入時一次性轉換為 equipment 武器條目（equipped=true）。
- 替代方案：保留 weapons 清單與物品欄雙軌——同一把武器存在兩處必然失同步，否決。

### D7. UI 結構沿用現有語彙
- 物品頁分區改為：已裝備（weapon/armor 且 equipped）／攜帶中（其餘）／任務物品（quest 置頂徽章或獨立分組）
- 新增入口：物品區塊標題列 trailing `SectionEditIcon` → bottom sheet 二選一（從目錄挑選／自訂輸入），沿用 `EditorSheetScaffold`
- 目錄挑選：手機單欄清單＋搜尋；平板（≥600dp）主從式（左清單右詳情），符合 Material 3 與現行 ResponsiveLayout 慣例
- 數量調整/使用/裝備切換直接在物品列上操作（觸控目標 ≥48dp）；刪除走左滑（比照日誌卡片），任務物品加確認

## Risks / Trade-offs

- [舊資料的 `type` 字串與新 `itemType` 並存造成混淆] → `type` 明確定位為顯示副標；行為一律看 `itemType`，程式內不再讀 `type` 做判斷
- [內容庫 items 資料未匯入前 App 端功能空轉] → 目錄挑選入口在 items 資料域為空時降級隱藏，自訂輸入不受影響；匯入屬內容庫專案作業，tasks 中列為前置
- [分層扣款換零規則實作錯誤導致財富損失] → 扣款邏輯獨立為純函式（`Currency payFrom(Currency, int priceCp)`）加單元測試覆蓋邊界（剛好足額、跨多階換零、ep 奇數換算）
- [雲端同步整包 JSON，物品增多後 payload 變大] → 現階段可接受（純文字），與既有法術清單同級；不做增量同步
- [AC 推導與舊靜態值不一致造成使用者困惑] → 推導引擎純函式化並以 mock 角色（法師無甲 12、野蠻人無甲防禦）驗證與現值一致；差異即為 bug
- [自訂武器缺機制資料無法推導命中/傷害] → 自訂武器編輯 sheet 提供傷害骰/屬性欄位（選填），缺值時攻擊列顯示名稱不顯示數值

## Migration Plan

1. 模型欄位全帶 `@Default`，舊 JSON 反序列化自然補值（source 預設 custom、quantity 預設 1）——無需資料修復腳本
2. 內容庫先行：items 匯入管線（比照既有資料域，`srd52` 過濾）→ 驗證筆數後 App 端才開目錄入口
3. 回滾：App 端 revert 即可；新欄位留在雲端 JSON 不影響舊版讀取（freezed 忽略未知欄位）

## Open Questions

- SRD 5.2 裝備清單的中譯來源與涵蓋筆數需在匯入時盤點（武器/護甲/冒險裝備/工具各多少條）——影響 catalog-source 涵蓋需求的驗收數字
- 「使用」消耗品要不要記錄到冒險日誌（自動寫一筆「使用了治療藥水」）——本次先不做，觀察需求
